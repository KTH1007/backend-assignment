# 프로젝트를 진행하면서 중요하게 생각한 부분
## 1. 할인 정책의 유연한 확장
요구사항에서 할인 정책은 수시로 추가 및 변경이 가능하다고 명시되어 있었습니다.
이 부분에서 변경에 유연하게 대응할 수 있는 구조가 필요하다고 생각했습니다.

DiscountPolicy 인터페이스를 정의하고 각 등급별 정책을 구현체로 분리했습니다.
DiscountPolicyResolver가 Spring DI를 활용해 모든 구현체를 자동으로 수집하기 때문에,
새로운 등급이 생기더라도 기존 코드를 전혀 수정하지 않고 새로운 클래스만 추가하면 되서 OCP를 자연스럽게 지킬 수 있는 구조라고 생각했습니다.

## 2. 정적 팩토리 메서드와 private Builder
엔터티 생성시 @Builder를 private로 제한하고 create() 정적 팩토리 메서드만 외부에 노출했습니다.
Builder를 public으로 열어두게 되면 외부에서 불완전한 상태의 객체를 만들 수 있다는 문제가 있습니다.
create() 메서드를 통해서만 객체를 생성하도록 강제해서 객체가 항상 유효한 상태로 생성되는 것을 보장하고 생성 의도를 명확하게 드러낼 수 있다고 생각했습니다.

## 3. 금융 계산에서 double 사용 지양
처음에는 자연스럽게 double을 사용했지만, 부동소수점 특성상 금액 계산에서 오차가 발생할 수 있다는 것을 인지했습니다.
금융 도메인에서 1원의 오차도 허용되지 않기 떄문에 정수 연산으로 변경했습니다.

## 4. 동시성 문제
중복 결제를 체크하더라도 두 요청이 동시에 들어오면 문제가 발생할 수 있습니다. 결제처럼 데이터 정확성이 중요한 경우에는 낙관적 락보다는 비관적 락이 적합하다고 판단했습니다.

## 5. Clock 주입을 통한 테스트 용이성
LocalDateTime.now()를 직접 사용하면 테스트에서 시간을 제어할 수 없어 검증이 어려울 수 있습니다. Clock을 빈으로 등록하고 주입받아서 사용하면 시간을 고정할 수 있어
예측 가능한 테스트 작성이 가능합니다.

## 6. DDD에서 영감을 받은 구조
비즈니스 요구사항이 자주 바뀌는 환경을 고려해서 도메인 중심으로 패키지를 구성했습니다. 각 도메인이 독립적으로 변경될 수 있도록 member, order, discount 등으로 분리하고
각 도메인 내에서는 계층을 두었습니다.

## 7. 공통 응답 및 공통 예외 처리
API 응답 포맷이 일관되지 않으면 클라이언트에서 처리하기 어렵습니다. ApiResponse<T>로 모든 응답을 통일하고, 예외 처리도 한 곳에서 관리했습니다.
ErrorCode enum이 HTTP 상태코드와 메시지를 함께 관리하기 때문에 새로운 에러 추가 시 enum만 추가하면 되도록 만들었습니다.

# main 브랜치에서 feature 브랜치로 넘어가면서 변경 및 개선한 코드의 논리적 근거
## 1. 할인 이력 비정규화 및 불변성 보장
main에는 할인 정책을 변경하면 과거 결제 내역의 할인 근거가 사라지는 문제가 있었습니다. 이를 해결하기위해 DiscountHistory에 정책명, 할인율, 할인금액을 문자열 + 정수로 저장했습니다.
또한 모든 이력 필드에 updatable = false를 적용하여 저장 이후 수정 자체를 DB레벨에서 차단했습니다.

## 2. 결제 수단 할인에 전략 패턴 도입
main에서는 등급 할인에만 전략 패턴을 적용했습니다. feature에서 포인트 중복 할인을 추가할 때 단순 if문으로 조건 분기로 구현할 수 있었지만, 새로운 결제 수단이 추가될 때마다
PaymentService를 추가해야 돼서 OCP 위반이 발생할 수 있습니다. 등급 할인과 동일한 패턴으로 전략 패턴을 도입해 서비스단에서는 결제 수단 종류를 몰라도 되는 구조로 개선했습니다.

## 3. DiscountPolicy 인터페이스 확장
이력 저장 시 할인율이 필요했는데 VIP는 고정금액 정책이라 rate가 원금에 따라 달라졌습니다. 각 정책이 자신의 할인율을 직접 계산하도록 getDiscountRate(int originalPrice)를
인터페이스에 추가했습니다.
